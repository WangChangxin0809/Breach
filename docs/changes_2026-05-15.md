## 二、同 Party 分配到同一阵营（未提交）

**问题**：匹配成功后服务端按加入顺序交替分配阵营（`assignFaction`），同一房间成员可能被打散。

**范围**：`server/modules/state/match.go`、`server/modules/match_handler.go`

### 改动点

**MatchState 新增字段**

```go
PartyFaction map[string]Faction  // party_id → 分配的阵营
UserParty    map[string]string   // user_id → party_id
PartySize    map[string]int      // party_id → 人数
```

**AssignFaction 方法**：`AssignFaction(userID) Faction`
- 查 `UserParty` 确认归属 party
- 若该 party 已分配过阵营，直接返回 → 同一 party 所有人同阵营
- 首次分配时按当前攻守人数平衡选择，锁定到 `PartyFaction`

**matchmakerMatched**：匹配成功时按 `entry.GetPartyId()` 分组，通过 `party_groups` 参数传入 `MatchCreate`

**MatchInit**：解析 `party_groups`，填充 `UserParty` / `PartySize`

**MatchJoin**：`assignFaction(current)` → `current.AssignFaction(userID)`

**删除**：旧的全局 `assignFaction()` 函数

## 三、匹配状态广播到全体房间成员（未提交）

**问题**：房主点击"开始匹配"后，只有房主本地进入 `MATCHMAKING` 状态锁定按钮，其他成员仍可操作。

**范围**：`client/scripts/main_game/config.gd`、`network_client.gd`、`room.gd`

### 改动点

**新增 `PARTY_OP_MATCHMAKING := 2`**（config.gd）

**新增 `send_room_matchmaking(active: bool)`**（network_client.gd）
- 通过 PartyData 广播 `{"matchmaking": true/false}`

**`_on_room_data_received`**（room.gd）
- 前置处理 `PARTY_OP_MATCHMAKING`：收到后调用 `_enter_state` 锁定或解锁按钮
- 取消准备触发的退出匹配追加广播 `send_room_matchmaking(false)`

**触发时机**：
| 触发点 | 广播值 | 说明 |
|--------|--------|------|
| `_on_start_match_pressed` | `true` | 房主点击开始匹配 |
| `_cancel_matchmaking` | `false` | 任何人取消 / 房主取消准备 |
| `_on_room_data_received` 取消准备 | `false` | 非房主取消准备时通知所有人 |
| `_on_network_status_changed` 失败 | `false` | 匹配器返回失败 |
