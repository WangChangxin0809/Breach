# 后端配置三层架构

后端配置建议分为三层：**代码常量层、JSON 玩法配置层、数据库/运营配置层**。这样可以兼顾性能、可维护性和未来运营扩展。

## 第一层：代码常量层

用于稳定、低频变化、偏技术性质的值。

适合放在 Go 代码中：

- match module name
- op code
- protobuf protocol version
- 安全硬上限
- 内部状态枚举

这类值通常和代码协议、服务结构强绑定，不适合让策划表或后台随意修改。

示例：

```go
const (
    OpCodeMove      int64 = 1
    OpCodeGameState int64 = 2
)
```

## 第二层：JSON 玩法配置层

用于大多数可调玩法参数，是当前项目的主要配置来源。

适合放在 `server/modules/config/data/`：

- 角色属性
- 武器属性
- 技能冷却和效果
- 回合时长
- 地图尺寸和障碍物
- 击杀奖励
- 解锁费用

优点：

- 不需要在逻辑代码里硬编码玩法数值。
- 配置结构清晰，方便比较、审查和版本管理。
- Go 启动时加载成强类型结构，运行时只读内存，不影响 MatchLoop 性能。
- 可以通过 `go:embed` 打进服务插件，部署稳定。
- 可以通过 `BREACH_CONFIG_DIR` 在本地或服务器覆盖。

当前实现已经采用这一层作为主要方案：

```text
server/modules/config/data/match.json
server/modules/config/data/characters.json
server/modules/config/data/weapons.json
```

游戏逻辑通过 `config.Active()` 读取配置。

## 第三层：数据库/运营配置层

用于后期运营、后台管理、赛季配置和玩家持久化数据。

适合存入 PostgreSQL 或 Nakama storage：

- 玩家背包
- 角色/武器解锁状态
- 段位、战绩、货币
- 赛季配置版本
- 后台发布的活动配置
- 灰度配置

不建议在高频战斗逻辑中直接查询数据库，尤其不要在 `MatchLoop` 每 tick 访问数据库。推荐做法是：

1. 后台或数据库保存配置版本。
2. 服务启动或创建 match 时加载配置。
3. 校验配置。
4. 将配置快照放入内存。
5. 单局对局只读取这个内存快照。

这样可以保证同一局对战中的规则一致，也避免数据库延迟影响战斗手感。

## 推荐使用原则

- 技术协议和稳定标识放代码常量。
- 玩法调数放 JSON。
- 玩家数据、赛季版本、后台运营数据放数据库。
- 战斗循环只读内存配置，不直接查数据库。
- 配置必须有版本号和启动校验。
- 单局 match 应使用同一份配置快照，避免对局中途规则变化。

## 当前阶段建议

当前项目仍处于核心玩法开发期，推荐优先完善第二层 JSON 配置：

- 补全角色配置。
- 补全武器配置。
- 增加技能配置。
- 增加经济和解锁配置。
- 给配置添加更完整的测试。

等后台管理、赛季系统、玩家解锁系统成熟后，再引入第三层数据库/运营配置。不要过早把所有玩法参数都放进数据库，否则会增加部署、调试和一致性成本。
